- Move the mouse deliberately, don't rush between screens

- Pre-navigate to the pages before the recording so they load fresh

- Don't use jargon like "CRUD" or "entity" — speak in plain language

## Security — Key Talking Points

### Authentication
- **Form-based login** — Username (email) and password submitted to `/login`
- **BCrypt password hashing** (strength 10) — Passwords are never stored in plain text
- **Single session enforcement** — Only one active session per user at a time, preventing session sharing
- **30-minute session timeout** — Automatic logout after inactivity
- **Session invalidation on logout** — `JSESSIONID` cookie is deleted and the HTTP session is destroyed

### Authorisation (Role-Based Access Control)
Four distinct roles control what each user can see and do:

| Role | URL Access | What They Can Do |
|------|-----------|-----------------|
| **centre-admin** | `/admin/**` | Full CRUD on players, users, badges, modules, teams, age groups |
| **coach** | `/coach/**` + `/player/**` | Approve/reject evidence submissions, view player profiles |
| **player** | `/player/**` | View own profile, submit evidence for challenges |
| **parent** | `/player/**` | View their child's progress and badge status |

### Additional Security Controls
- **CSRF protection** enabled on all state-changing operations (forms include CSRF tokens)
- **Clickjacking prevention** — Frame options set to `SAMEORIGIN`
- **URL-level access control** — Spring Security filter chain enforces role checks before any page renders
- **No public API surface** — All data access requires authentication

## Data Model — What We Store and Why

### Core Entities (16 JPA tables)

**Users & Access:**
- `app_user` — Email, encrypted password, name, profile image
- `role` — centre-admin, coach, parent, player
- `user_role` — Maps users to roles *per centre* (supports multi-centre, multi-role)

**Gamification Engine:**
- `badge` — 5 persistent badges (Game Mastery, Teamwork, Esports Citizen, Personal Development, Digital Skills)
- `sub_badge` — Individual challenges with XP values (10–20 XP each)
- `skill` — 10 youth work outcomes (Teamwork, Communication, Leadership, etc.)
- `level` — XP thresholds: Bronze (0–30), Silver (31–70), Gold (71–120), Platinum (121–200), Diamond (201+)

**Player Progression:**
- `player` — Links a user to a centre, age group, team, current module; tracks `totalXp`
- `player_badge_detail` — Individual challenge completion: evidence URL/file, approved flag, coach notes, earned date
- `player_level_detail` — Player's current level per badge with accumulated `badgeXp`

**Learning Structure:**
- `module` — Named programmes (e.g., "Defeat the Ender Dragon", "Road to Diamond — Rocket League")
- `module_detail` — Weekly session plans with learning outcomes, linked challenges, session plan/slide links
- `team` — Player groupings (Minecraft Juniors, Rocket League Group, etc.)
- `age_group` — Age brackets (Juniors 8–14, Seniors 13–18, Competitive 16–25)

## Business Benefits

### 1. Eliminates Manual Admin Work
- **Before:** Hours spent updating spreadsheets, WordPress, and Google Sheets manually
- **After:** One system handles player tracking, badge progression, and session planning — all updated in real time
- **Impact:** Youth workers reclaim time to spend with young people instead of on admin

### 2. Real-Time Visibility for All Stakeholders
- **Admins** see a live dashboard of the entire academy
- **Coaches** have a clear approval queue — no emails or WhatsApp chasing
- **Players** see their own progress instantly — badges, XP, leaderboard position
- **Parents** can log in and see exactly what their child is working on and achieving

### 3. Structured Evidence & Accountability
- Every challenge completion is backed by submitted evidence (URL or file upload)
- Coach approval creates an auditable record of achievement
- No more "he said she said" — digital proof tied to every badge earned

### 4. Gamification Drives Engagement
- XP points, badge levels (Bronze → Diamond), and a public leaderboard create healthy competition
- Young people can see exactly what they need to do next to level up
- Challenges are bite-sized and tied to weekly sessions — achievable and motivating

### 5. Progressive Web App (PWA) — Works Everywhere
- Runs on any device with a browser — phone, tablet, laptop, Chromebook
- No app store submission, no installation required
- Responsive design with accessibility-first approach (ARIA labels, semantic HTML)
- Offline-capable architecture ready for areas with patchy connectivity

### 6. Multi-Centre / SaaS-Ready Architecture
- The `Centre` entity scopes all data — players, badges, modules, and teams belong to a centre
- Users can hold different roles at different centres
- This means any YMCA (or similar organisation) could run their own instance from the same platform
- **Licensing model:** One codebase, many centres — scalable without code changes

### 7. Safeguarding & Data Protection
- No personal data exposed publicly — players only see their own profile
- Role-based access means coaches can't access admin functions and players can't see other players' submissions
- Session management prevents shared logins
- All passwords are hashed — even a database breach would not expose credentials

### 8. Legacy Data Migration Support
- `LegacyPoint` entity exists specifically to import historical data from the old spreadsheet system
- Ensures no young person loses progress when the new system goes live

## Potential Q&A Answers

**"Why H2 and not PostgreSQL/MySQL?"**
> H2 is a zero-config embedded database — perfect for a hackathon demo. The app uses Spring Data JPA, so switching to PostgreSQL or MySQL in production is a one-line config change. No code changes needed.

**"Is this production-ready?"**
> The architecture is production-ready — layered services, proper security, transactional data access. For a live deployment, you'd swap H2 for a managed database, add HTTPS/TLS, and configure external session storage. The app design doesn't need to change.

**"How does it handle multiple YMCAs?"**
> Every entity is scoped to a `Centre`. The `UserRole` join table maps users to roles per centre. This means you could onboard a second YMCA tomorrow without touching the code — just add a new Centre and its users.

**"What about accessibility?"**
> ARIA labels on all interactive elements, semantic HTML roles, dark theme for readability, responsive viewport, form validation with clear error messaging. Designed to be usable on any device by any user.

## Production Readiness — Technical Debt

| # | Item | Notes |
|---|------|-------|
| 19 | **Swap H2 for PostgreSQL/MySQL** | H2 is fine for demo but not for production. Spring Data JPA makes this a config-only change. |
